# Go/No-Go Checklist (Pilot -> Full GMP ERP/MES)

Цель: подтвердить, что пилотная версия строится как production-grade foundation и масштабируется до full версии без переписывания ядра.

Как использовать:
- Для каждого пункта поставить статус: `GO`, `PARTIAL`, `NO-GO`.
- Если `PARTIAL` или `NO-GO`, открыть action item с датой и ответственным.
- Старт разработки разрешать только при отсутствии `NO-GO` по критичным пунктам (1-10).

## 1) Конфигурируемая модель сайта и процессов
**Проверка:** процессы завода (Process 1-7) заведены как конфигурация, а не hard-code.
**Почему критично:** иначе multi-site/full версия потребует рефакторинга доменной модели.
**Доказательство:** схема сущностей + конфиги site PQS/SMK map.
**Критерий GO:** можно добавить второй сайт без изменения кода ядра.

## 2) Единое GMP-ядро прав доступа
**Проверка:** роли, департаменты, permissions централизованы и переиспользуются во всех модулях.
**Почему критично:** локальные ACL по модулям приводят к расхождению и переделке.
**Доказательство:** RBAC матрица + интеграционные тесты на доступ.
**Критерий GO:** одна роль используется в EDMS/WMS/QC/EBR без дублирования логики.

## 3) Electronic Signature как платформенный сервис
**Проверка:** e-signature не "вшита" в отдельные формы, а вызывается единообразно.
**Почему критично:** при расширении модулей не должно быть 10 разных реализаций подписи.
**Доказательство:** единый API/сервис подписи + журнал событий.
**Критерий GO:** все критичные действия подписываются через общий механизм.

## 4) Audit Trail как immutable слой
**Проверка:** фиксируются old/new value, user, timestamp, reason; события нельзя тихо удалить.
**Почему критично:** без этого полный CSV и инспекции приведут к переделке.
**Доказательство:** audit trail спецификация + негативные тесты.
**Критерий GO:** критичные GxP-сценарии покрыты positive/negative тестами.

## 5) Workflow Engine вместо hardcoded state machine
**Проверка:** ключевые потоки (документы, QA release, deviation/CAPA) параметризуются workflow-движком.
**Почему критично:** full версия добавит десятки новых workflows.
**Доказательство:** минимум 3 workflow-конфигурации из пилота работают без изменения кода ядра.
**Критерий GO:** новая ветка согласования добавляется конфигом.

## 6) Канонические статусы качества
**Проверка:** `quarantine/under test/released/blocked/rejected/expired` реализованы как общий словарь.
**Почему критично:** локальные статусы по модулям ломают сквозной контроль.
**Доказательство:** единая модель quality status + правила переходов.
**Критерий GO:** QC/QA решения управляют складскими статусами, а не ручное редактирование.

## 7) Batch/Lot traceability end-to-end
**Проверка:** материал -> производство -> выпуск -> склад ГП связываются через lot genealogy.
**Почему критично:** без этого full traceability и recall потребуют перепроектирования.
**Доказательство:** отчет трассировки вверх/вниз по цепочке.
**Критерий GO:** для любой серии видны все входящие lot и исходящие партии.

## 8) EBR как шаблонный движок шагов
**Проверка:** EBR строится через template + step types + mandatory gates.
**Почему критично:** форма "на один продукт" не масштабируется.
**Доказательство:** 1 пилотный шаблон и возможность клонировать под второй маршрут.
**Критерий GO:** оператор не может пропустить обязательный шаг.

## 9) Разделение master data и transactional data
**Проверка:** справочники (materials/specs/equipment/SOP versions) отделены от операций.
**Почему критично:** смешение данных усложняет change control и миграции.
**Доказательство:** ERD + правила версионирования master data.
**Критерий GO:** изменение спецификации не ломает исторические записи.

## 10) Версионирование SOP/форм и привязка к исполнению
**Проверка:** каждый workflow/EBR шаг фиксирует версию SOP/формы на момент исполнения.
**Почему критично:** иначе при ревизии документов теряется юридическая/регуляторная прослеживаемость.
**Доказательство:** controlled record view/PDF с версией документа.
**Критерий GO:** obsolete версия не может стартовать новый процесс.

## 11) URS -> FRS -> Test Traceability Matrix
**Проверка:** требования из pilot matrix трассируются в FRS и тест-кейсы.
**Почему важно:** без трассировки full validation превращается в ручной аудитный долг.
**Доказательство:** живой traceability matrix по пилотному vertical slice.
**Критерий GO:** нет orphan-URS и orphan-test.

## 12) Data Integrity и correction model
**Проверка:** исправления делаются addendum-подходом без перезаписи оригинала.
**Почему важно:** ALCOA+ и инспекционные требования.
**Доказательство:** сценарии correction/late entry/backdating control.
**Критерий GO:** finalized GMP запись нельзя изменить без следа и обоснования.

## 13) Integration contract-first
**Проверка:** интерфейсы между WMS/QC/EBR/QA оформлены как контракт (events/API), не "прямыми" вызовами БД.
**Почему важно:** облегчает масштабирование и замену модулей.
**Доказательство:** спецификация событий/endpoint + контрактные тесты.
**Критерий GO:** модуль QC можно развивать независимо от WMS.

## 14) Migration and reference data strategy
**Проверка:** есть план начальной загрузки справочников/остатков/документов и стратегия валидации миграции.
**Почему важно:** в full rollout миграции часто главный источник риска.
**Доказательство:** migration playbook + dry-run результаты.
**Критерий GO:** повторяемая миграция на тестовой среде проходит без ручных правок.

## 15) Operability: backup, restore, monitoring, periodic review
**Проверка:** есть процедуры backup/restore, журнал техсобытий, контроль производительности, periodic review.
**Почему важно:** production-grade определяется не UI, а управляемой эксплуатацией.
**Доказательство:** SOP/инструкции + тест восстановления.
**Критерий GO:** подтвержденный restore в целевое RTO/RPO.

---

## Правило финального решения

**GO для старта пилотной разработки:**
- Пункты 1-10: только `GO` (без исключений).
- Пункты 11-15: допускается до 2 статусов `PARTIAL`, но с утвержденным планом закрытия.

**NO-GO (старт запрещен):**
- Любой `NO-GO` в пунктах 1-10.
- Отсутствие трассировки URS/FRS/Test по критичным GxP-сценариям.
- Отсутствие подтвержденных механизмов e-signature/audit trail.

## Минимальный пакет артефактов до кодинга

1. Утвержденный pilot SOP-to-system matrix.
2. URS v0.1 по vertical slice.
3. FRS v0.1 (ядро + pilot flow).
4. Traceability Matrix v0.1.
5. Data model draft (core objects + связи).
6. RBAC matrix.
7. e-signature policy.
8. audit trail specification.
9. Validation strategy (IQ/OQ/PQ подход).
10. План миграции стартовых справочников и документов.
