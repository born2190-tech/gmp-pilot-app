# Electronic Signature Policy v0.1 (Pilot Vertical Slice)

Дата: 2026-05-10  
Статус: Draft for QA/IT approval  
Связь: URS-004, URS-009, URS-016, URS-019, URS-021, URS-022

## 1. Цель

Определить единые правила применения электронных подписей для GxP-критичных действий в пилотной системе.

## 2. Базовые принципы

1. Электронная подпись однозначно связывается с конкретным пользователем и конкретной записью/действием.
2. Подпись содержит значение подписи (meaning), дату/время сервера и контекст действия.
3. Подпись невозможна без успешной повторной аутентификации пользователя.
4. Подпись является неотделимой частью controlled record и хранится в неизменяемом виде.

## 3. Обязательные элементы подписи

Каждый e-signature event включает:
- `signature_event_id`
- `user_id`
- `role_at_time`
- `action_type`
- `signature_meaning`
- `object_type` и `object_id`
- `batch_or_lot_reference` (если применимо)
- `timestamp_utc`
- `auth_method` (password re-entry или approved equivalent)
- `result` (`success`/`failed`)
- `reason/comment` (если требуется политикой)

## 4. Допустимые значения подписи (signature meaning)

- `Prepared by`
- `Reviewed by`
- `Approved by`
- `Released by QA`
- `Blocked/Rejected by`
- `Executed by`
- `Verified by`
- `Override approved by`

## 5. Какие действия требуют обязательной подписи

1. Перевод документа в `effective` или `obsolete`.
2. Коррекция finalized GMP-record.
3. Закрытие controlled record/EBR.
4. QA release decision (`release/block/reject`).
5. Старт production order при критичных pre-start checks.
6. Подтверждение критичных шагов EBR (operator/checker).
7. Закрытие deviation/CAPA/change этапов, где требуется QA approval.
8. Любой approved override (в т.ч. issue из недопустимого статуса при разрешении).

## 6. Правила dual-signature

Для пилота dual-signature требуется минимум для:
- `Override approved by` в сценариях URS-013 и URS-019.
- `Released by QA` для batch/lot с отклонениями, закрытыми в тот же период.

Модель dual-signature:
1. Первый подписант: процессный владелец (инициатор/ответственный).
2. Второй подписант: QA approver.
3. Пользователь 1 и пользователь 2 не могут совпадать.

## 7. Проверки перед подписью

До фиксации подписи система должна проверить:
- активность учетной записи;
- наличие требуемой роли/прав;
- актуальность training/qualification для критичной операции;
- целостность объекта подписи (нет конфликтующих изменений);
- отсутствие открытых блокирующих задач workflow.

## 8. Поведение при ошибках

- Неуспешная повторная аутентификация: подпись отклоняется, действие не выполняется.
- Недостаточно прав: подпись отклоняется, создается audit trail событие отказа.
- Изменение записи после открытия диалога подписи: требуется повторный запуск подписи.

## 9. Минимальные тесты e-signature

- ESIG-T01: успешная подпись QA release с корректным meaning и record binding.
- ESIG-T02: неверный пароль на re-auth -> действие не выполнено.
- ESIG-T03: попытка подписи без нужной роли -> отказ + audit event.
- ESIG-T04: dual-signature с одним и тем же пользователем -> отказ.
- ESIG-T05: изменение записи между review и signature -> требуется повторная подпись.

## 10. Решения для утверждения

1. Финальный перечень операций с dual-signature в пилоте.
2. Допустимые методы re-auth (только пароль или эквивалент).
3. Политика блокировки учетной записи после неудачных попыток подписи.
